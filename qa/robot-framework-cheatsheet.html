Robot Framework Cheat Sheet (Basic → Advanced)

Topic-wise reference for building production-grade Robot Framework automation frameworks: structure, design, scaling, CI, and reliability.

Use this as a framework checklist: keep tests readable, keep keywords layered, keep execution stable, and keep CI diagnostics rich.
Project structure File sections & settings Keyword design patterns Variables & scoping IF/FOR & control flow UI automation (selectors & waits) API automation Test data strategy Parallel execution (Pabot) Listeners & artifacts Reporting & CI integration Quality gates (Robocop)

1) Project structure (framework-level)

Recommended layout for large teams:

tests/
  ui/
  api/
resources/
  pages/       # page-level keywords + locators
  flows/       # business flows (domain language)
  common/      # reusable primitives (waits, utils)
libs/          # python libraries (API clients, parsers)
variables/     # env configs (NO secrets in git)
ci/            # pipelines/scripts
  • Tests call flows (business intent), flows call pages/services, pages call library keywords.
  • One responsibility per keyword (actions vs assertions vs data).

2) File sections & key settings

*** Settings ***
Library    SeleniumLibrary    timeout=10
Resource   ../resources/flows/Auth.resource
Suite Setup    Prepare Suite
Test Setup     Prepare Test
Test Teardown  Cleanup Test

*** Variables ***
${BASE_URL}    https://example.com

*** Test Cases ***
Login
    Login As    alice    secret
    Dashboard Should Be Visible

*** Keywords ***
Prepare Suite
    Log    Suite start
  • Suite Setup: expensive shared init (containers, seeded data).
  • Test Setup: per-test isolation (fresh user/session).

3) Keyword design patterns

  • Layering: UI primitives → Page keywords → Flow keywords → Tests.
  • Pure vs impure keywords: keep data builders pure; UI/API actions impure.
  • Assert where it matters: after state-changing actions, assert stable outcomes.
*** Keywords ***
Login As
    [Arguments]  ${user}  ${pass}
    Input Text    ${LOGIN_USER}    ${user}
    Input Password ${LOGIN_PASS}   ${pass}
    Click Button   ${LOGIN_SUBMIT}
    Wait Until Page Contains    Dashboard

4) Variables & scope

  • Prefer test/suite variables over global to stay parallel-safe.
  • Use dicts for structured test data.
Set Test Variable     ${ORDER_ID}    ${id}
Set Suite Variable    ${TOKEN}       ${token}

5) Control flow (RF 4+)

IF    ${status} == 200
    Log    OK
ELSE
    Fail   Unexpected status
END

FOR    ${item}    IN    @{ITEMS}
    Log    ${item}
END

Use Wait Until Keyword Succeeds only for eventual consistency, not as a flake bandaid.

6) UI automation (stability checklist)

  • Selectors: prefer data-testid, avoid brittle XPath.
  • Waits: wait for readiness conditions; avoid Sleep.
  • Isolation: each test owns its user/data.
Wait Until Element Is Visible    css=[data-testid="ready"]    10s
Click Element                    css=[data-testid="submit"]

7) API automation (RequestsLibrary)

*** Settings ***
Library    RequestsLibrary

*** Test Cases ***
Health
    Create Session    api    https://example.com
    ${r}=    Get On Session    api    /health
    Should Be Equal As Integers    ${r.status_code}    200
  • Keep auth/session in setup; assert status + schema + key fields.

8) Test data strategy

  • Create data via API per test; tag with run id.
  • Cleanup in teardown (idempotent cleanup).
  • For shared pools (rare), lock allocation only.
[Teardown]    Delete Order Via Api    ${ORDER_ID}

9) Parallel execution (Pabot)

  • No global shared state, no shared download dirs, no shared users.
  • Separate outputs per run/shard.
pabot --processes 4 --outputdir results-parallel tests/

10) Listeners & artifacts

Use listeners to attach screenshots, logs, and extra metadata for fast triage.

class Listener:
    ROBOT_LISTENER_API_VERSION = 3
    def end_test(self, data, result):
        if not result.passed:
            # take screenshot / collect logs
            pass

11) Reporting & CI

  • Always upload log.html, report.html, output.xml as CI artifacts.
  • Generate JUnit for CI test dashboards.
rebot --xunit junit.xml --outputdir results output.xml

12) Quality gates (Robocop + conventions)

  • Enforce naming, line lengths, complexity, duplication checks.
  • Use code reviews + templates.
robocop --reports all tests/
