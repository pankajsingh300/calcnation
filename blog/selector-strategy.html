XPath vs CSS Selectors: Which Should You Use?

📅 February 15, 2026 📂 Automation Best Practices ⏱️ 6 min read

Bad selectors cause flaky tests. Good selectors are stable, readable, and maintainable. This guide covers both CSS selectors and XPath, helping you choose the right strategy for your tests.

CSS Selectors

Advantages

  • Faster than XPath (browser-optimized)
  • Simpler syntax
  • Works across all browsers
  • Easier to read and maintain

Examples

/* ID selector (most specific, best) */
#login-button

/* Class selector */
.submit-btn

/* Attribute selector */
[data-test-id="login"]

/* Pseudo-selector */
button:nth-child(2)

/* Descendant combinator */
.form .input-field

/* Adjacent sibling */
label + input

Best Practice CSS Selectors

// ❌ Brittle: dependent on DOM structure
button:nth-child(5) > span

// ✅ Better: uses stable attributes
[data-test-id="submit-button"]

// ❌ Too generic
input[type="text"]

// ✅ Better: combines attributes for uniqueness
input[type="text"][name="username"]

XPath

Advantages

  • More powerful and flexible
  • Can traverse up the DOM (parent/ancestor)
  • Text matching
  • Complex conditions

Examples

/* Find by text */
//button[contains(text(), 'Login')]

/* Find by attribute */
//input[@id='username']

/* Find parent */
//input[@name='password']/parent::form

/* Find next sibling */
//label[text()='Email']/following-sibling::input

/* Complex conditions */
//button[@type='submit' and contains(@class, 'primary')]

When to Use XPath

  • Matching by text content
  • Traversing up to find parent elements
  • Complex conditional logic
  • Finding elements without stable IDs/classes

CSS Selectors vs XPath: Comparison

Criterion CSS XPath
Speed Fast Slower
Traverse up (parent) ❌ No ✅ Yes
Text matching ❌ Limited ✅ Yes
Readability ✅ Easy ⚠️ Complex

Avoiding Flaky Selectors

❌ Bad Practices

  • Relying on element position (nth-child)
  • Generic selectors (.button, input[type="text"])
  • Full DOM paths
  • Hardcoding generated class names
  • Text-based selectors without fallbacks

✅ Good Practices

  • Use data-test-id attributes (specifically for QA)
  • Prioritize IDs if they're stable
  • Use semantic HTML (label/input relationships)
  • Combine multiple attributes for uniqueness
  • Keep selectors as specific as possible

Recommendation

Default to CSS selectors for speed and simplicity. Use XPath only when you need:

  • Text matching
  • Parent/ancestor traversal
  • Complex conditions

In Playwright, you can also use text-based selectors:

// Playwright text selector
await page.click('button:has-text("Login")')

// Or with test-id
await page.click('[data-test-id="login-button"]')
