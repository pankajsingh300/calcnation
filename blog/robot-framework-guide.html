Robot Framework Guide for QA Automation Engineers

📅 January 15, 2026 📂 Automation Testing ⏱️ 8 min read

Robot Framework is a keyword-driven automation framework that makes test automation accessible and maintainable. Unlike code-heavy tools like Selenium or Playwright, Robot Framework uses readable English-like syntax, making tests readable by both technical and non-technical stakeholders.

What is Robot Framework?

Robot Framework is an open-source test automation framework that uses a simple, keyword-based syntax. It's perfect for:

  • Acceptance testing
  • Web application automation
  • API testing
  • Mobile app testing
  • RPA (Robotic Process Automation)

Installation

Robot Framework requires Python 3.6+:

pip install robotframework
pip install robotframework-seleniumlibrary  # For web automation
pip install robotframework-requests          # For API testing

Basic Syntax

Robot Framework uses .robot files with a keyword-driven approach:

*** Settings ***
Library    SeleniumLibrary

*** Test Cases ***
Login Test
    Open Browser    https://example.com    chrome
    Input Text    id:username    admin
    Input Text    id:password    password123
    Click Button    id:login-btn
    Page Should Contain    Dashboard
    Close Browser

*** Keywords ***
User Login
    [Arguments]    ${username}    ${password}
    Input Text    id:username    ${username}
    Input Text    id:password    ${password}
    Click Button    id:login-btn

Key Concepts

Test Cases

Test cases are defined under *** Test Cases *** section. Each test case is a series of keywords (built-in or custom).

Keywords

Keywords are reusable blocks of logic. They can be built-in (from libraries) or custom (defined by you).

Variables

Variables are defined with ${variable_name}:

*** Variables ***
${BASE_URL}       https://example.com
${USERNAME}       admin
${PASSWORD}       password123

*** Test Cases ***
Login
    Open Browser    ${BASE_URL}    chrome
    Input Text    username    ${USERNAME}

Libraries

Libraries extend Robot Framework's functionality. Common ones:

  • SeleniumLibrary — Web browser automation
  • RequestsLibrary — HTTP API testing
  • DatabaseLibrary — Database operations
  • Telnet Library — Network testing

Web Automation Example

*** Settings ***
Library    SeleniumLibrary

*** Test Cases ***
E2E Checkout Flow
    Open Browser    https://shop.example.com    chrome
    Click Link    Electronics
    Click Element    xpath://button[@data-id='laptop-123']
    Click Button    Add to Cart
    Click Link    Cart
    Click Button    Checkout
    Input Text    email    user@example.com
    Input Text    card-number    4111111111111111
    Click Button    Purchase
    Page Should Contain    Order Confirmed

API Testing Example

*** Settings ***
Library    RequestsLibrary

*** Test Cases ***
GET User API
    ${response}=    GET    https://api.example.com/users/1
    Should Be Equal As Integers    ${response.status_code}    200
    Should Contain    ${response.text}    "id": 1

POST Create User
    ${json}=    Create Dictionary    name=John    email=john@example.com
    ${response}=    POST    https://api.example.com/users    json=${json}
    Should Be Equal As Integers    ${response.status_code}    201

Best Practices

1. Use Data-Driven Testing

Parameterize tests with multiple inputs:

*** Test Cases ***
Login With Various Credentials
    [Template]    Verify Login
    admin          password123    Dashboard
    user           pass456        User Home
    guest          guest789       Limited Access

*** Keywords ***
Verify Login
    [Arguments]    ${user}    ${pass}    ${expected_page}
    Open Browser    https://example.com    chrome
    Input Text    username    ${user}
    Input Text    password    ${pass}
    Click Button    Login
    Page Should Contain    ${expected_page}
    Close Browser

2. Create Custom Keywords

Encapsulate complex workflows into reusable keywords for cleaner tests.

3. Use Page Object Model (POM)

Separate test logic from UI element locators to make tests maintainable.

4. Organize Tests by Feature

Group related tests into test suites by feature or module.

5. Use Descriptive Names

Test names should clearly describe what's being tested:

✅ Good: User Can Successfully Login With Valid Credentials
❌ Bad: Test 1

Running Tests

robot test.robot                          # Run single file
robot tests/                                # Run all .robot files in directory
robot --tag login tests/                   # Run tests with specific tag
robot --variable BROWSER:firefox test.robot # Override variables

Comparison: Robot Framework vs. Playwright

  • Robot: Keyword-driven, readable, slower, large community
  • Playwright: Code-first, fast, powerful, modern DevOps focus

Choose Robot Framework for business-focused automation, Playwright for developer-focused automation.
